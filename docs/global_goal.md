软件工程综合实验要求

海外藏中国文物知识管理与服务平台

北京化工大学

二零二六年四月

实验要求学生综合运用软件工程相关知识，设计并实现一个面向海外藏中国文物的知识管理与服务平台。该平台以知识图谱技术为核心，整合数据爬取、知识建模、信息检索、可视化展示、智能问答等多项技术，涵盖Web端、移动端及后台管理系统的完整软件开发实践。

平台共分为五个子系统，如下所述：

- 知识图谱构建子系统：从海外博物馆网站爬取并建模中国文物数据；
- 海外文物知识服务子系统：提供Web端数据浏览、查询与可视化服务；
- 知识问答子系统：基于知识图谱实现智能问答功能；
- 掌上博物馆（移动端App）：面向移动用户的文物展示与交互功能；
- 后台管理子系统：对全平台数据与用户进行统一管理。

### 1、知识图谱构建子系统

编写程序从海外博物馆网站爬取中国文物信息，经数据清洗、建模与存储，构建结构化的海外文物知识图谱，为知识服务、问答及可视化功能提供数据基础。主要包括以下功能：

**（1）数据爬取**

爬取至少3家指定海外博物馆的全部中国文物信息（博物馆列表见附录），爬取内容至少包括：文物名称、文物图片（原图）、年代、类型、材质、介绍文本、文物详情页URL。数据按规定格式保存（具体格式见附录），以UTF-8编码输出为CSV文件，并提交字段说明文档。

技术要求：需处理反爬机制；图片需解析原始地址下载，不得仅保存缩略图；爬取完成后需统计各博物馆文物数量并核验数据完整性。

**（2）数据清洗与质量控制**

对爬取的原始数据进行清洗处理，主要包括：

- 字段标准化：统一年代格式（如"公元前206年"统一表示方式）、文物类型分类等；
- 去重处理：识别并合并同一文物的重复记录；
- 图片有效性验证：检测图片链接是否可正常访问，过滤无效图片。

**（3）数据建模**

将清洗后的数据转化为三元组形式（主体-谓词-客体），建议参考CIDOC-CRM文化遗产领域本体标准进行建模。至少需要定义以下实体类型和关系：

- 实体类型：文物（Artifact）、博物馆（Museum）、朝代（Dynasty）、艺术家（Artist）、地点（Location）等；
- 关系类型：收藏于、创作于、属于朝代、材质为、类型为等。

**（4）数据补充**

针对基础数据中存在的信息缺失，从外部来源进行定向补充：

- 从百度百科、维基百科等爬取信息，例如书画作家生平、朝代背景、文物介绍等信息；
- 补充数据须与原始数据建立关联，并标注来源与补充日期。

**（5）实体对齐与去重**

不同博物馆可能收藏同一艺术家的作品或记录相同的历史实体，需进行跨数据源的实体对齐：

- 对艺术家、朝代、地点等共享实体进行识别与合并；
- 建立实体唯一标识符（URI），避免知识图谱中出现重复节点；
- 对齐结果须可追溯，保留原始来源信息。

**（6）数据存储**

建立双层存储架构，分别满足图查询和业务查询需求：

- 图数据库（Neo4j 或 Virtuoso）：存储全部三元组，支持图查询、关系遍历与SPARQL检索；可选发布为链接开放数据（LOD）；
- 关系型数据库（MySQL 或其他）：存储文物详细数据、用户数据及系统业务数据，支持高效的结构化查询。

**（7）数据更新与增量爬取**

为保持知识图谱的时效性，系统应支持定期数据更新：

- 支持增量爬取，仅爬取上次爬取后新增或更新的文物记录，避免全量重复下载；
- 记录每次爬取的时间、数量与变更详情；
- 对更新的实体数据自动触发三元组重建与图数据库同步。

### 2、海外文物知识服务子系统

编写Web端程序，基于知识图谱构建子系统获取的数据，为用户提供文物数据的浏览、查询、可视化与个性化服务。界面设计可参考克利夫兰艺术博物馆、大英博物馆等国际一流博物馆网站。

**（1）数据浏览**

提供多维度、多形式的文物浏览体验：

- 列表与卡片视图：支持在列表视图与图片卡片视图之间切换，满足不同用户浏览习惯；
- 多维筛选与排序：支持按文物类型、年代、所属博物馆、材质等多个维度进行组合筛选，支持按名称、年代、更新时间等字段排序；
- 文物详情页：完整展示文物的文本描述、高清图片、三元组属性等信息；图片支持缩放、全屏查看；显示文物在知识图谱中的关联实体；
- 相关文物推荐：在详情页展示相关文物，推荐规则可基于相似主题、同一作者、同一朝代、图像内容相似度等，推荐算法自定；
- 文物对比功能：支持用户选择2—3件文物进行横向属性对比，直观展示差异。

**（2）数据查询**

提供从简单到复杂的多层次查询能力：

- 简单查询：支持按文物名称、博物馆名称、年代等关键字进行全文检索，结果页支持分页与二次筛选；
- 高级查询：支持对文物类型、年代范围、材质、收藏地等多个字段进行组合限定查询，可参考克利夫兰博物馆的高级搜索功能；
- 知识图谱查询：支持用户以自然语言或结构化方式查询实体间的关系，例如"查询某朝代的所有瓷器"或"查询某位艺术家的全部作品"；
- 查询结果导出：支持将查询结果导出为CSV或JSON格式，便于用户进一步研究使用。

**（3）数据可视化**

从多个维度对文物知识图谱进行可视化展示：

- 知识图谱关系图：采用力导向图展示文物实体及其关联关系，支持节点点击展开、悬停显示属性、缩放与拖拽交互，可参考全历史网站、历史人文大数据平台等；
- 文物时间轴：以时间轴形式展示各历史时期的文物分布，支持按朝代或自定义时间段筛选，点击时间节点可跳转至对应文物列表；
- 文物地理分布图：以世界地图形式展示文物现藏博物馆的地理位置，支持点击地图标注查看该博物馆的藏品信息，直观呈现中国文物的海外流散现状
- 统计分析看板：以图表形式展示文物数量、类型占比、朝代分布、博物馆藏量等统计数据，为研究者提供宏观数据视角。

**（4）用户个人信息管理**

- 基础功能：用户注册与登录，支持设置用户名、密码、头像等个人信息；
- 浏览记录（选做）：自动记录用户最近浏览的文物，支持一键清除；
- 收藏功能（选做）：支持用户收藏感兴趣的文物，收藏夹支持分组管理；
- 评论与互动（选做）：支持用户对文物发表评论，支持点赞与回复；
- 个性化推荐（选做）：根据用户浏览与收藏记录，在首页推送可能感兴趣的文物。

### 3、知识问答子系统

本子系统基于知识图谱数据与大语言模型，为用户提供自然语言形式的文物知识问答服务，支持从简单属性查询到复杂关系推理的多层次问答能力。

建议采用"知识图谱+大语言模型"的检索增强生成（RAG）架构：

- 用户输入自然语言问题；
- 系统进行意图识别与实体抽取，将问题转化为图数据库查询语句（SPARQL或Cypher）；
- 从知识图谱中检索相关事实与数据；
- 将检索结果作为上下文传入大语言模型，生成自然流畅的回答；
- 返回答案并标注数据来源。

该架构既保证了答案的事实准确性（来自知识图谱），又具备良好的语言表达能力（来自大语言模型），有效避免大模型幻觉问题。

**（1）简单问答**

支持单实体单属性的问答，至少覆盖以下10类问题：

- 文物收藏地：该文物现藏于哪家博物馆？
- 文物年代：该文物属于哪个历史时期？
- 文物材质：该文物由什么材料制成？
- 文物类型：该文物属于哪种器物类别？
- 文物介绍：请介绍一下这件文物。
- 书画作者：该书画作品的作者是谁？针对书画类文物。
- 作者生平：该作者的生平经历是怎样的？针对书画类文物。
- 同一作者作品：该作者还有哪些作品被海外博物馆收藏？
- 同一朝代文物：某朝代有哪些代表性文物？
- 文物尺寸与规格：该文物的尺寸和重量是多少？
- 相关文物推荐：与该文物风格或主题相似的文物有哪些？

技术要求：对于知识图谱中不存在的问题，应明确告知用户"暂无相关数据"，而非生成不实答案。

**（2）答案溯源与可信度展示**

为增强问答系统的可信度，每条回答须附带数据来源说明：

- 标注答案来源（如"数据来源：克利夫兰艺术博物馆"）；
- 提供指向原始数据详情页的链接；
- 对于由大语言模型生成的补充性描述，须与知识图谱事实性内容明确区分标注。

**（3）多轮对话（选做）**

支持基于上下文的连续追问，提升问答交互的自然性：

- 系统能够理解代词指代，例如用户先问"青花瓷是什么？"后问"它主要产于哪个朝代？"，系统能正确理解"它"指代青花瓷；
- 保留最近5轮对话上下文用于意图理解；
- 支持用户主动切换话题，系统能识别并重置上下文。

**（4）复杂问答（选做）**

支持需要多步推理或跨实体关联的复杂问题，主要包括：

- 多跳推理问答：例如"收藏青铜器最多的博物馆是哪家？该博物馆在哪个城市？"
- 比较类问答：例如"唐代和宋代的瓷器在风格上有何不同？"
- 统计类问答：例如"大都会博物馆共收藏了多少件中国文物？"
- 路径查询：例如"某件文物与某位历史人物之间有什么关联？"

技术建议：可基于知识图谱的多跳查询（如SPARQL路径查询）结合大语言模型进行推理链生成。

**（5）问答质量反馈机制（选做）**

建立用户反馈，持续优化问答质量：

- 每条回答下方提供"有帮助"与"不准确"两个反馈按钮；
- 对于用户标记为不准确的回答，记录至问题日志供人工审核；
- 统计高频未能正确回答问题类型，为知识图谱补充与模型优化提供依据。

**（6）闲聊与文化导览（选做）**

支持与用户进行文物文化相关主题的自由对话：

- 能够介绍中国各历史朝代的文化背景与艺术特色；
- 能够讲解中国传统工艺（如陶瓷烧制、书画技法等）的基础知识；
- 支持根据用户兴趣推荐相关文物或拓展阅读内容；
- 对于超出文物文化领域的话题，礼貌告知用户系统专注范围。

### 4、掌上博物馆

基于HarmonyOS开发手机端App，建议使用ArkTS语言与ArkUI框架进行开发。从知识图谱数据中选取单类文物作为展示主题，例如瓷器、玉器、青铜器或中国画等，为用户提供沉浸式的文物浏览与互动体验。可参考故宫博物院App（https://www.dpm.org.cn/Creative.html#app）。

**（1）文物浏览**

提供丰富的文物展示与浏览体验：

- 首页展示：以卡片或瀑布流形式展示文物列表，支持按热度、年代、类型等排序；
- 文物详情：展示文物高清图片（支持手势缩放与多图切换）、基本属性、详细介绍文本；
- 音视频讲解：若提供讲解音频或视频，支持在文物详情页内嵌播放，支持倍速与进度控制；
- 简单搜索：支持按文物名称、朝代、关键字进行全文搜索，搜索结果支持分类展示。

**（2）语音导览**

利用HarmonyOS的语音能力，提供解放双手的文物导览体验：

- 为每件文物提供标准普通话语音讲解，支持在文物详情页一键播放；
- 支持语音搜索：用户可通过语音输入关键字查找文物；
- 支持语音问答：用户可对着手机说出关于文物的问题，系统调用知识问答子系统返回语音答案。

**（3）用户交互**

丰富的用户参与以及内容共创功能：

- 点赞与收藏：支持用户对文物点赞和收藏，收藏夹支持分组管理；
- 留言评论：支持用户对文物发表文字评论，支持点赞与回复他人评论；评论须经内容审核后显示；
- 用户上传照片：用户可上传自己拍摄的文物相关照片（如参观博物馆时拍摄），照片须经审核后公开展示；支持为照片添加拍摄地点与简短说明；
- 用户上传讲解（选做）：支持用户录制或上传针对某件文物的音频讲解，经审核后可供其他用户收听；
- 用户上传视频（选做）：支持用户上传与文物相关的短视频，经审核后展示。

**（4）以图搜图**

支持基于图像特征的文物检索功能：

- 图片上传搜索：用户从相册选择图片上传，系统提取图像特征后在文物数据库中检索相似文物；
- 实时拍照搜索：用户直接调用摄像头拍摄文物或相关图片，即时返回匹配结果；
- 搜索结果展示：按相似度排序展示匹配文物列表，显示相似度分值与文物基本信息；
- 技术建议：可使用预训练的图像特征提取模型（如ResNet、ViT等）提取文物图像特征向量，存入向量数据库，搜索时计算余弦相似度进行检索。

**（5）用户个人信息管理**

- 注册与登录：支持手机号、邮箱注册，支持密码登录；可选支持华为账号一键登录；
- 个人主页：展示用户的收藏、点赞、上传内容、浏览历史等个人数据；
- 隐私设置：用户可控制个人信息的可见范围；
- 用户系统可与海外文物知识服务子系统共用，或单独建立，需在设计文档中说明选择理由。

**（6）用户动态（选做）**

- 支持用户发布包含文字与图片的动态，分享文物或博物馆相关内容：
- 动态可关联到具体文物或博物馆，点击可跳转至对应详情页；
- 其他用户可对动态点赞与评论；
- 支持关注其他用户，首页展示关注用户的最新动态；
- 所有动态内容须经过内容审核方可公开展示。

**（7）消息通知（选做）**

- 利用HarmonyOS通知能力，向用户推送个性化消息：
- 当用户上传的照片或评论通过审核后，发送通知；
- 当关注的用户发布新动态时，发送通知；
- 定期推送"文物今日推荐"，引导用户探索新内容；
- 用户可在设置中自定义通知类型与频率注意。

可自行设计增加更多功能。可参考https://www.dpm.org.cn/ Creative.html#app。

### 5、后台管理子系统

编写Web端后台管理程序，对平台全部数据、用户、内容与系统运行状态进行统一管理，确保平台安全、稳定、高效运行。后台系统须设置独立的管理员账号体系，与前台用户账号分离。

**（1）角色与权限管理**

建立基于角色的访问控制机制，至少定义以下三类角色：

- 超级管理员：拥有全部操作权限，包括创建和管理其他管理员账号、系统配置等；
- 内容审核员：仅拥有内容审核相关权限，无法操作用户数据与系统配置；
- 数据管理员：拥有文物数据的增删改查权限，无法操作用户账号与系统日志。

支持自定义角色，可灵活分配各模块的查看、编辑、删除等细粒度权限；所有权限变更须记录操作日志。

**（2）用户管理**

对全平台所有用户进行统一管理，涵盖后台管理员、知识服务子系统用户、掌上博物馆用户：

- 用户信息管理：支持对用户基础信息的增删改查，支持按用户名、注册时间、用户来源等条件筛选，支持批量操作；
- 用户状态管理：支持启用、禁用用户账号；对违规用户支持细粒度权限限制，如仅禁止评论、禁止上传图片等，而非直接封禁账号；
- 用户行为记录：可查看特定用户的操作历史，包括发布内容、评论、上传记录等，便于违规行为溯源；
- 违规处理记录（选做）：记录每次对用户的处罚操作，包括处罚原因、操作人、处罚时间与解除时间，支持申诉处理流程。

**（3）内容审核**

建立"自动审核+人工审核"的双重内容审核机制，确保平台内容合规：

自动审核：

- 文本审核：维护敏感词库，支持管理员在线增删敏感词；用户提交内容时自动检测，命中敏感词则拦截并提示用户修改；
- 图片审核：调用图像识别接口对用户上传图片进行违规内容检测，审核不通过的图片自动屏蔽并记录；
- 自动审核结果可配置：管理员可设置各类内容的审核策略，如"低风险自动通过、高风险转人工审核"。

人工审核：

- 待审核队列：展示所有待人工审核的内容（文本、图片、音视频），支持按内容类型、提交时间筛选；
- 审核操作：审核员可对每条内容执行通过、拒绝、标记复审等操作，拒绝时须填写拒绝原因；
- 批量审核：支持对同类内容进行批量通过或拒绝操作，提升审核效率；
- 审核统计：展示每日审核量、通过率、拒绝率等统计数据，支持按审核员维度统计工作量。

**（4）数据管理**

对全平台所有业务数据进行统一管理：

- 文物数据管理：对知识图谱中的文物数据进行增删改查，支持批量导入（CSV/JSON格式）与导出，支持图片的批量上传与替换；
- 知识图谱管理：支持对三元组数据的在线编辑，包括添加、修改、删除实体与关系，修改后可触发图数据库同步更新；
- 用户生成内容管理：管理用户上传的评论、图片、音视频等内容，支持按来源子系统、内容类型、审核状态筛选；
- 数据一致性检查（选做）：定期检测关系型数据库与图数据库之间的数据一致性，对发现的不一致记录生成报告并支持手动修复。

**（5）数据备份与恢复**

保障平台数据安全，防止数据意外丢失：

- 手动备份：管理员可随时触发全量备份或指定数据库的备份操作；
- 定时自动备份：支持配置定时备份任务，如每日凌晨自动全量备份，备份文件自动加密存储；
- 备份记录管理：展示所有备份记录列表，包括备份时间、备份类型、文件大小、操作人等信息；支持下载备份文件；
- 数据恢复：支持从任意备份记录点恢复数据库，恢复前须经超级管理员二次确认；恢复操作须记录详细日志；
- 备份存储管理：支持设置备份保留策略（如保留最近30天的备份），自动清理过期备份文件。

**（6）日志管理**

建立完善的操作日志体系，支持审计与问题排查：

- 操作日志：记录所有管理员的后台操作，包括操作人、操作时间、操作类型、操作对象与操作前后的数据变化；
- 系统日志：记录系统运行中的异常事件、接口错误、定时任务执行情况等；
- 安全日志：记录登录行为（包括失败登录）、权限变更、敏感数据访问等安全相关事件
- 日志查询：支持按时间范围、操作人、操作类型、关键字等多维度检索日志；
- 日志导出：支持将查询结果导出为CSV或Excel格式，便于合规审计。

**（7）系统监控看板**

实时展示平台运行状态，帮助管理员及时发现并处理异常：

- 实时数据统计：展示当前在线用户数、今日新增用户数、今日内容提交量、审核队列积压量等关键指标；
- 访问量统计：以折线图展示各子系统的日/周/月访问量趋势；
- 数据增长统计：展示文物数据量、用户数量、内容数量等随时间的增长趋势，为运营决策提供数据支撑；
- 异常告警（选做）：当系统出现接口响应超时、数据库连接失败、磁盘空间不足等异常时，在看板显示告警信息并支持邮件或短信通知管理员。

**（8）系统配置管理（选做）**

支持在线修改系统运行参数，无需重启服务：

- 敏感词库管理：在线增删改查敏感词，支持批量导入；
- 审核策略配置：配置各类内容的自动审核阈值与人工审核触发条件；
- 系统公告管理：发布、编辑、下线面向前台用户的系统公告或维护通知；
- 功能开关：支持对各子系统的特定功能进行在线开启或关闭，便于灰度发布与紧急处置。

### 附录2：数据爬取格式及要求

**（1）数据格式要求**

所有爬取数据须遵循以下格式规范：

- 编码格式：统一使用 UTF-8 编码保存，避免中文乱码；
- 文件格式：保存为 CSV 格式，建议使用 Python 的 csv 或 pandas 库进行读写，防止出现分隔符冲突、换行符异常等格式错误；
- 文件命名：按博物馆命名，例如 cleveland_museum.csv、metropolitan_museum.csv；
- 图片存储：图片文件按博物馆分文件夹存放，文件名与 CSV 中的文物 ID 一一对应，例如 images/cleveland/obj_12345.jpg；
- 各组须自行验证导出的 CSV 文件可在 Excel 或 pandas 中正常打开，确认无乱码、无格式错误后方可提交。

**（2）数据字段要求**

每条文物记录至少包含以下字段（部分字段如博物馆网站未提供可标注为空）：

| **字段名（英文）** | **中文说明** | **是否必填** | **备注** |
| --- | --- | --- | --- |
| object_id | 文物唯一标识符 | 必填  | 使用博物馆原始ID或自行生成 |
| title | 文物名称 | 必填  | 优先保留英文原名，有中文译名则补充 |
| period | 年代/时期 | 必填  | 如"Tang Dynasty"或"618–907 AD" |
| type | 文物类型 | 必填  | 如 Painting、Ceramics、Bronze 等 |
| material | 材质  | 建议填写 | 如 Silk、Bronze、Jade 等 |
| description | 文物介绍 | 必填  | 爬取网站提供的原始描述文本 |
| dimensions | 尺寸  | 建议填写 | 如"H. 30 cm × W. 20 cm" |
| museum | 所属博物馆 | 必填  | 填写博物馆完整英文名称 |
| location | 博物馆所在地 | 必填  | 城市、国家 |
| detail_url | 文物详情页URL | 必填  | 文物在博物馆网站上的原始页面链接 |
| image_url | 图片原始下载链接 | 必填  | 须为原图地址，而非缩略图 |
| image_path | 本地图片存储路径 | 必填  | 图片下载后的相对路径 |
| credit_line | 版权/来源说明 | 建议填写 | 博物馆提供的版权声明 |
| accession_number | 藏品编号 | 建议填写 | 博物馆馆藏编号 |
| crawl_date | 爬取日期 | 必填  | 格式：YYYY-MM-DD |

注意事项：

- 图片爬取须注意解析原图地址，部分网站（如大都会博物馆）需要通过API获取高清原图链接，不可仅保存页面展示的缩略图；
- 对于缺失字段，用空字符串填充，不可省略该列；
- 文物介绍若为英文，建议同时保留原文，后续可通过翻译API补充中文描述。

**（3）数据量要求**

- 每组须爬取指定的3家博物馆，筛选其中的中国文物；
- 最终提交数据总量不少于5000件文物的有效记录（含图片）。

**（4）数据质量验收标准**

提交前须自行检查以下内容，并在说明文档中附上检查结果：

- 必填字段完整率：object_id、title、detail_url、image_url、crawl_date 五个必填字段不得有空值；
- 图片有效性：随机抽查20条记录，验证图片文件可正常打开，且为原图而非缩略图；
- 编码正确性：读取 CSV 后打印前5行，确认中文字符无乱码；
- URL 有效性：随机抽查10条 detail_url，验证链接可正常访问。

**（5）说明文档要求**

每组须提交一份数据说明文档（Markdown 或 Word 格式），内容包括：

- 字段说明表：列出所有字段的英文名、中文含义、数据类型与示例值，例如：

| **英文字段名** | **中文说明** | **数据类型** | **示例** |
| --- | --- | --- | --- |
| period | 文物年代 | 字符串 | Tang Dynasty |
| crawl_date | 爬取日期 | 日期  | 2024-03-15 |

- 爬取说明：说明各博物馆爬取时遇到的技术难点及解决方案，如反爬处理方式、原图地址解析方法等
- 数据统计：各博物馆爬取记录数、图片下载成功率、字段缺失情况统计
- 数据质量自查报告：按照第（4）条验收标准的自查结果
